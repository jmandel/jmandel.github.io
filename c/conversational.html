1) Make Policies & Evidence Accessible

Today: Criteria buried in PDFs, portals, and “tribal knowledge”.

Better: Publish the same logic reviewers rely on—human-readable and machine-tractable.

// Ideal: share the actual decision logic { "policy": "adalimumab_ra", "criteria": { "diagnosis": "rheumatoid_arthritis", "prior_therapy": { "drug": "methotrexate", "duration": ">= 3 months", "outcome": "failure_or_intolerance" }, "safety_screening": ["TB_test"] } }

If it's good enough for internal decision-making, it's the most useful external representation.

3) Nudge vs. Sludge

✅ Nudge (Good Friction)

  • Conservative care first

⛔ Sludge (Bad Friction)

  • Hidden requirements
  • Redundant forms
  • Arbitrary delays

Goal: Maximize nudges, eliminate sludge.

MCP: Make EHR Data LLM-Friendly

Problem: LLMs drown in massive FHIR bundles.

Solution: Give general-purpose tools (search, evaluate) instead.

// Don't stream 10MB of JSON. Provide tools: tools: [ { name: "grep", description: "Search all patient data", parameters: { pattern: "string", context_lines: "number" } }, { name: "eval_js", description: "Run JavaScript on patient data", parameters: { code: "string" } } ] // Usage examples: grep("methotrexate") // finds mentions in notes, med list, etc. eval_js(` observations .filter(o => o.code === 'blood_pressure') .map(o => o.value) .reduce((sum, v) => sum + v) / observations.length `)

14

Liaison: Smart Agent Mediation

Pattern: Decide when to involve the user vs. handling it automatically.

// Decision examples // Case 1: Handle automatically { "action": "respond_automatically", "reason": "Can retrieve lab date from EHR via grep tool", "proposed_response": "Found TB test from 2024-03-15" } // Case 2: Ask for focused input { "action": "ask_user", "reason": "Multiple medications match 'methotrexate'", "user_prompt": "Which methotrexate formulation did the patient try?", "options": ["Oral tablets", "Injectable", "Both"] }

Result: minimal interruption, maximum automation.

15

Liaison: Detect & Structure State Changes

Pattern: Listen for important state changes in varied formats and normalize them for the app.

// Examples the Liaison normalizes // BCBS: "AUTH GRANTED #8847221 EXP:05/24/26" // Aetna: PDF approval details on page 3 // United: "Approval Code PA2024-78432 (see attached)" // Consistent app update: app.updateStatus({ "type": "approval_received", "approval_number": "PA-2024-78432", "valid_through": "2026-05-24", "drug": "adalimumab", "conditions": ["Use biosimilar if available"], "user_message": "✅ Prior auth approved! Valid for 12 months." })

Putting It All Together

Scenario: Dr. Reed orders adalimumab for David Lee (RA)

1. EHR agent checks requirements via payer’s A2A endpoint

2. Payer agent returns specific criteria

3. EHR agent uses MCP tools to gather evidence from FHIR APIs

4. Iterative dialogue clarifies missing pieces

5. Liaison surfaces only critical decisions to the clinician

6. Approval in minutes—not days

Same clinical decision, far less administrative burden.
