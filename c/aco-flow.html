Overview

This explainer was automatically generated from a discussion transcript between Josh Mandel and Anna Taylor. It describes a networked FHIR architecture for ACO quality reporting and population level clinical data exchange.

Scope: Quality reporting for attributed patients at population scale. Start with in network providers, extend to outside providers within a trust framework.

Problem to solve

ACOs need timely, complete clinical data across many providers to compute measures and close care gaps. Today this often relies on manual chart chase workflows with CSV or PDF and ad hoc agreements. The objective is a repeatable, standards based pipeline that scales to tens of thousands of patients while honoring policy and consent.

Standards and artifacts

Da Vinci ATR for rosters and Group formation. FHIR Bulk Data export for USCDI data in NDJSON. OAuth2 and mTLS within the trust fabric. Provenance and AuditEvent for traceability.

Architecture overview

The ACO runs core services that discover participants, negotiate groups, apply consent policy, perform exports, and compute measures. External parties are payers and providers connected through a shared trust fabric.

In network happy path

  1. Payer sends attribution roster to the ACO.
  2. ACO proposes roster to each participating provider using ATR and receives a confirmed group id.
  3. ACO requests bulk export for the group with USCDI resources.
  4. Provider returns accepted with status link. ACO polls and downloads NDJSON when ready.
  5. ACO normalizes data and computes measures, then drives gap closure.

Outside network targeted path

  1. ACO ranks external providers by claims overlap and selects targets.
  2. ACO attaches machine readable consent artifacts when required.
  3. ACO proposes a limited roster with ATR and provider may approve after review.
  4. ACO requests bulk export for the approved group and ingests results.

Key technical decisions

  • Population first not broadcast target providers by claims signal and work down the list.
  • Group scoped exports always request by group id per counterparty.
  • Incrementals use export since after initial full pull.
  • Identity matching prefer deterministic keys and record linkage provenance.
  • Data minimization request only needed resource types for measures.
  • Deep observability capture purpose of use consent id group id token scope and timings.

Challenges and mitigations

Consent complexity

Central store for consent with machine readable policy. Attach evidence to ATR and export. Record decisions with audit events.

Patient versus population

Define an operations profile under the same trust rails with tailored token scopes and request content.

Data quality

Normalize codes and track source quality scores. Fill gaps with mappings where safe.

Scale

Parallelize by source and resource type with safe retries and resume. Use queues to manage back pressure.

Roster drift

Refresh ATR on cadence and on claims deltas. Handle joins and leaves idempotently.

Human in the loop

Support manual approval paths without breaking pipeline automation.

Security privacy and compliance

  • Mutual tls and oauth with short lived tokens and least privilege scopes.
  • Key and certificate lifecycle tied to the trust fabric with automated rotation.
  • Encryption at rest careful phi logging controls and segregated environments.
  • Complete audit trail of requester subject policy consent and payloads.
  • Data retention aligned to contracts and state law.

Roadmap

  • Standardize operations population profile under the trust framework with clear scopes.
  • Extend ATR to carry consent references and policy qualifiers.
  • Interoperable consent with provenance revocation and partial sharing.
  • Measure aware export presets that pull only needed elements.
  • Network level discovery of supported resources history depth and rate limits.
